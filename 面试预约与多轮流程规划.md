# 面试预约与多轮流程规划（含短信提醒）

> 文档目标：先把业务逻辑定清楚，作为产品/前后端评审基线；本版不涉及代码实现细节。  
> 版本：v1（MVP 规划）  
> 日期：2026-02-13

## 1. 目标与范围

### 1.1 业务目标
- 对“拟面试人员”进行一面/二面/三面预约管理。
- 每轮面试结束后可决策：进入下一轮 / 通过 / 淘汰 / 待定。
- 在关键节点自动发送短信提醒应聘者。
- 全流程可追踪（谁在什么时候改了什么）。

### 1.2 本期范围（MVP）
- 支持最多三轮面试（轮次 1~3）。
- 支持预约、改期、取消、记录结果。
- 支持短信发送：预约确认、改期通知、取消通知、面试前提醒。
- 支持终态控制：通过、淘汰后不可继续创建新轮次。

## 2. 核心对象与术语

- 候选人流程（主流程）：候选人当前整体进度。
- 面试轮次（子流程）：第 N 轮面试的具体安排与结果。
- 通知任务：一次待发送/已发送短信任务。
- 终态：`已通过`、`已淘汰`（终态后流程锁定）。

## 3. 状态机设计

## 3.1 候选人主状态
- `待安排`
- `已预约`
- `面试中`
- `待下一轮安排`
- `待定`
- `已通过`（终态）
- `已淘汰`（终态）
- `已放弃`（终态，可选）

## 3.2 面试轮次状态
- `草稿`
- `待通知`
- `已通知`
- `候选人已确认`（可选）
- `进行中`
- `已完成`
- `已取消`
- `缺席`

## 3.3 每轮面试结果（完成时必选其一）
- `进入下一轮`
- `通过`
- `淘汰`
- `待定`

## 3.4 主流程流转规则（简化）
1. 新加入拟面试池 -> `待安排`
2. 创建第 N 轮预约成功 -> `已预约`
3. 到达面试时间/开始面试 -> `面试中`
4. 记录第 N 轮结果：
- `进入下一轮` -> `待下一轮安排`
- `通过` -> `已通过`（终态）
- `淘汰` -> `已淘汰`（终态）
- `待定` -> `待定`

## 4. 业务规则（必须先定死）

1. 终态规则：主状态为 `已通过`/`已淘汰`/`已放弃` 后，禁止新建轮次。  
2. 唯一进行中轮次：同一候选人同一时刻只能有一个“有效未完成轮次”。  
3. 结果唯一：每轮面试只能有一个最终结果。  
4. 进入下一轮前置：必须先完成当前轮并选择 `进入下一轮`。  
5. 改期规则：仅未完成且未取消的轮次可改期。  
6. 取消规则：取消后不可直接记录该轮结果；需重新安排新轮次。  
7. 缺席规则：可标记缺席，后续允许“再次预约同轮补面”或直接结束（由业务决定）。  
8. 幂等规则：同一事件短时间内不可重复触发短信（防重发）。  
9. 审计规则：所有状态变化必须记录操作人、时间、变更前后值。  

## 5. 短信触发矩阵（MVP）

| 触发事件 | 发送对象 | 发送时机 | 模板建议 | 备注 |
|---|---|---|---|---|
| 面试预约成功 | 候选人 | 立即 | `INTERVIEW_BOOKED` | 包含时间、地点/链接、联系人 |
| 面试改期成功 | 候选人 | 立即 | `INTERVIEW_RESCHEDULED` | 明确“原时间 -> 新时间” |
| 面试取消 | 候选人 | 立即 | `INTERVIEW_CANCELLED` | 包含后续联系提示 |
| 面试前提醒 | 候选人 | 面试前 24 小时（MVP 先一个） | `INTERVIEW_REMINDER_24H` | 后续可加 2 小时提醒 |

### 5.1 失败重试（建议）
- 重试 3 次：1 分钟、10 分钟、30 分钟。
- 超过重试次数转“人工处理”，在管理端告警。

### 5.2 合规与风控（建议）
- 仅在发送时间窗内发送（如 08:00-21:00）。
- 模板变量做非空校验（时间、地点、联系人、电话）。
- 记录供应商回执码与失败原因，便于追查。

## 8. 权限建议

- HR 专员：可安排、改期、记录结果、发送短信。
- HR 主管：具备 HR 专员权限，且可回退状态（受限场景）。
- 超级管理员：全局可见、可纠错、可强制终态恢复。
- 普通用户不可直接修改终态记录。

## 9. 异常场景处理

1. 候选人电话无效：短信失败进入人工处理队列。  
2. 面试官临时变更：走“改期/改人”操作并自动发改期短信。  
3. 已通过后误操作：仅主管以上可“撤销终态”，且必须写原因。  
4. 重复点击安排：后端需幂等校验，避免重复建轮次/重复发短信。  
5. 跨时区问题：系统统一存 UTC，展示按本地时区。  

## 10. 分期落地建议

### Phase 1（先上线）
- 一~三轮预约与结果记录。
- 4 类短信触发。
- 基础日志审计。

### Phase 2
- 候选人确认到面。
- 二次提醒（T-2h）。
- 更细的面试评价维度与评分。

### Phase 3
- 数据看板：到面率、轮次转化率、平均排期时长、短信送达率。
- 自动化规则（如待安排超时提醒 HR）。

## 11. 评审清单（开会直接对齐）

1. 主状态是否精简到可执行？  
2. 每轮结果四选一是否满足业务？  
3. 终态是否允许回退？谁有权限？  
4. 缺席后是补约还是直接结束？  
5. 短信模板文案与触发时机是否确认？  
6. MVP 是否只保留 24h 提醒，先不上 2h 提醒？  

## 12. 后端接口清单（草案）

> 约定：以下接口均为规划草案，路径风格与现有 `api/admin/*` 保持一致。  
> 鉴权：沿用现有 Token 机制。  
> 幂等：创建预约与发送短信建议支持 `Idempotency-Key` 头。

### 12.1 候选人流程与轮次

1. 获取候选人面试流程概览  
- `GET /api/admin/interview-processes/`  
- 查询参数：`status`、`region_id`、`job_id`、`keyword`、`page`、`page_size`  
- 返回：候选人主状态、当前轮次、下一场时间、短信状态摘要

2. 获取候选人流程详情  
- `GET /api/admin/interview-processes/{application_id}/`  
- 返回：主流程 + 全部轮次时间线 + 短信记录摘要

3. 创建/预约某一轮面试  
- `POST /api/admin/interview-rounds/`  
- 请求体建议：
```json
{
  "application_id": 123,
  "round_no": 1,
  "scheduled_start_at": "2026-03-01T10:00:00+08:00",
  "scheduled_end_at": "2026-03-01T11:00:00+08:00",
  "interview_mode": "onsite",
  "location_or_link": "北京总部A座301",
  "interviewer_ids": [8, 15],
  "contact_name": "李老师",
  "contact_phone": "13800001111",
  "send_sms_now": true
}
```
- 返回：轮次记录、主状态更新结果、短信任务创建结果

4. 改期  
- `PATCH /api/admin/interview-rounds/{round_id}/reschedule/`  
- 请求体：`scheduled_start_at`、`scheduled_end_at`、可选备注  
- 返回：更新后的轮次信息、短信触发结果

5. 取消该轮  
- `POST /api/admin/interview-rounds/{round_id}/cancel/`  
- 请求体：`reason`、`send_sms_now`  
- 返回：取消结果、短信触发结果

6. 记录轮次结果  
- `POST /api/admin/interview-rounds/{round_id}/result/`  
- 请求体建议：
```json
{
  "result": "next_round",
  "score": 82,
  "feedback_summary": "基础不错，建议二面",
  "tags": ["沟通良好", "化工经验"]
}
```
- `result` 枚举：`next_round` / `pass` / `reject` / `pending`
- 返回：轮次状态变更、主流程状态变更、可选下一步指引

7. 创建下一轮草稿（可选接口）  
- `POST /api/admin/interview-processes/{application_id}/next-round-draft/`  
- 前置条件：上一轮结果必须是 `next_round`

### 12.2 短信任务与日志

1. 手动补发短信  
- `POST /api/admin/notification-tasks/{task_id}/retry/`

2. 获取候选人短信日志  
- `GET /api/admin/interview-processes/{application_id}/notifications/`

3. 获取失败任务列表（运营视角）  
- `GET /api/admin/notification-tasks/?status=failed`

### 12.3 错误码建议

- `INTERVIEW_PROCESS_CLOSED`：候选人已终态，不可创建新轮次  
- `ROUND_ALREADY_COMPLETED`：该轮已完成，不能改期/取消  
- `INVALID_RESULT_TRANSITION`：非法结果流转  
- `SMS_TEMPLATE_DATA_INVALID`：短信模板变量不完整  
- `SMS_PROVIDER_FAILED`：短信通道失败  
- `NO_PERMISSION_REGION_SCOPE`：越权访问/操作

筛选项建议：

## 13.2 安排面试弹窗

字段：

- 开始/结束时间
- 面试方式
- 地点/会议链接

提交流程：
1. 前端校验时间合法性（结束 > 开始，不能预约过去时间）。
2. 调创建轮次接口。
3. 成功后刷新列表与候选人流程抽屉。

## 13.3 候选人流程抽屉（重点）

内容结构：
1. 主流程头部：当前状态、当前轮次、终态标签。  
2. 轮次时间线：一面/二面/三面每轮的状态、时间、面试官、结果。  
3. 短信记录：模板、发送时间、发送状态、失败原因。  
4. 操作区：改期、取消、记录结果、补发短信。

- 进入下一轮

附加输入：

- 标签（可选）

分支行为：
- 选“进入下一轮”后提示“是否立即安排下一轮”。
- 选“通过/淘汰”后主流程直接终态。

## 13.5 短信日志抽屉

表格字段：
- 事件类型
- 模板编码

- 发送状态
- 失败原因
- 重试次数

操作：
- 失败任务支持“立即重试”

## 14. 开发任务拆分（建议）

## 14.1 后端任务

1. 数据库迁移：新增流程、轮次、短信任务、操作日志表。  
2. 领域服务：状态机校验器（主状态与轮次状态）。  
3. API：完成第 12 节接口。  
4. 短信网关：通道适配、重试策略、回执落库。  
5. 权限：沿用区域权限并扩展到新接口。  
6. 审计：关键动作写操作日志。  

## 14.2 前端任务（管理端）

1. 拟面试列表升级（字段、筛选、操作入口）。  
2. 安排/改期/结果弹窗。  
3. 候选人流程抽屉（轮次时间线 + 短信日志）。  
4. 错误提示映射（后端错误码 -> 中文提示）。  
5. 失败重试交互。  

## 14.3 测试任务

1. 状态流转测试：合法/非法流转。  
2. 幂等测试：重复提交预约/短信。  
3. 权限测试：区域账号越权拦截。  
4. 短信失败重试测试。  
5. 回归测试：现有拟面试列表与应聘详情不回归。  

## 15. 验收标准（MVP）

1. 可完成“一面 -> 二面 -> 三面 -> 通过/淘汰”完整流程。  
2. 每轮结果必须四选一，且能正确驱动主流程状态。  
3. 预约/改期/取消会自动生成短信任务并可追踪状态。  
4. 终态后无法继续安排新轮次。  
5. 所有关键操作可在日志中追溯到操作人和时间。  
6. 区域权限生效，无越权读写。  

## 16. 实施顺序建议（两周示例）

第 1 周：
1. 冻结状态机与错误码。  
2. 完成后端表结构和核心接口（预约、结果、改期、取消）。  
3. 打通短信任务落库（先可不接真实通道）。  

第 2 周：
1. 管理端完成流程抽屉和操作弹窗。  
2. 接入短信真实发送 + 重试。  
3. 联调 + UAT + 生产开关灰度。  

## 17. 新增功能模块：面试评估卡片（参考应聘记录卡片）

> 目标：做一个和“应聘记录卡片”同层级的业务模块，既展示候选人应聘资料，也聚合展示面试过程与评估结果。  
> 备注：本节只做产品与交互规划，不涉及代码实现。

## 17.1 模块定位

- 在管理端新增 Tab：`面试评估卡片`（或在“拟面试人员”中增加视图切换：列表视图 / 卡片视图）。
- 用于 HR 在一屏内完成“候选人基本判断”：看资料、看轮次、看分数、看结论。
- 支持从卡片直接进入“安排面试 / 记录结果 / 查看完整详情”。

## 17.2 卡片信息架构（摘要层）

每张卡片建议固定显示：

1. 候选人基础信息
- 姓名、性别、年龄、手机号
- 应聘岗位、所属地区、招聘类型
- 头像（优先 `photo_url`，无图显示姓名首字）

2. 应聘资料摘要
- 最高学历
- 工作经历年限（可后续计算）
- 当前状态标签（待安排/进行中/待下一轮/通过/淘汰）

3. 面试评估摘要（核心新增）
- 当前轮次（一面/二面/三面）
- 最近一轮面试官（多人时显示 `张三 等2人`）
- 最近一轮综合评分（如 `82/100`）
- 最近一轮结果（进入下一轮/通过/淘汰/待定）
- 下一场面试时间（若已预约）

4. 短信摘要
- 最近一次短信状态（成功/失败/重试中）

5. 快捷操作
- `安排/改期`
- `记录结果`
- `查看全量档案`

## 17.3 详情抽屉（完整档案层）

点击卡片“查看全量档案”后打开右侧抽屉，分区如下：

1. 应聘资料区
- 复用现有应聘详情结构（基础信息、教育/工作/家庭、扩展字段）

2. 面试时间线区（新增）
- 轮次时间线（1~3 轮）
- 每轮字段：时间、方式、面试官、评分、评语、结果、操作人

3. 附件区（新增建议）
- 个人照片、身份证、简历、其他附件列表（文件名 + 上传时间）

4. 通知区
- 该候选人所有短信记录与发送状态

## 17.4 交互规则

1. 卡片选择支持批量（用于批量加入面试安排队列或批量导出）。  
2. 卡片主按钮随状态变化：
- `待安排`：主按钮为“安排一面”
- `进行中/待下一轮`：主按钮为“安排下一轮/记录结果”
- `终态`：主按钮为“查看结论”
3. 在卡片上记录结果时，必须弹窗二次确认（防误点）。  
4. 终态候选人卡片默认灰化部分操作按钮，仅保留查看与审计。

## 17.6 UI 方向（参考 ui-ux-pro-max）

基线风格建议：
- 主风格：`Data-Dense Dashboard`
- 布局：高信息密度卡片网格 + 右侧详情抽屉（summary -> detail）
- 交互：卡片 hover 高亮、行/卡片状态色、筛选切换动画 150~300ms

颜色建议（可落地为 CSS 变量）：
- `--card-primary: #0369A1`
- `--card-secondary: #0EA5E9`
- `--card-success: #22C55E`
- `--card-bg: #F0F9FF`
- `--card-text: #0C4A6E`

中文可读性建议：
- 标题/正文优先 `Noto Sans SC`（避免纯英文字体导致中文阅读体验下降）

质量约束（沿用 skill 清单）：
- 禁用 emoji 作为图标，统一 SVG 图标集
- 可点击卡片和按钮必须有明确 hover/focus 态
- 确保浅色模式文本对比度 >= 4.5:1
- 兼容 375/768/1024/1440 宽度

## 17.7 与现有模块关系

1. “应聘记录”保留：作为原始投递数据主入口。  
2. “面试评估卡片”新增：作为面试阶段的决策入口。  
3. 两者数据同源：应聘主记录 + 面试轮次 + 通知任务聚合，不重复维护两套数据。  

## 18. 对应任务补充（基于新模块）

后端补充：
1. 新增“评估卡片聚合列表”接口（可分页、可筛选）。  
2. 新增“候选人完整档案”接口（应聘资料 + 面试时间线 + 附件 + 通知）。  

前端补充：
1. 新增“面试评估卡片”页面或视图模式。  
2. 新增卡片详情抽屉（复用现有详情字段并扩展面试区/附件区）。  

验收补充：

2. 卡片信息与详情信息一致，不出现状态不一致问题。
